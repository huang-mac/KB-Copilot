# 智能知识库问答助手迭代文档

## 项目定位

本项目定位为一个面向中小企业的通用智能知识库问答系统，优先解决企业内部文档分散、检索效率低、知识复用困难的问题。

第一阶段不做 ERP/WMS 深度定制，而是先完成一个可运行、可演示、可部署、可写进简历的通用 RAG 项目。后续可以基于行业文档、业务字段和权限体系扩展到 ERP、WMS、客服、产品文档、企业制度等场景。

## 技术选型

| 模块 | 选型 | 说明 |
| --- | --- | --- |
| 前端 | React + Vite + TypeScript + Ant Design | 构建简洁的企业后台式界面，适合 GitHub 展示和简历包装 |
| 后端 | FastAPI | 提供文档上传、知识库管理、索引构建、问答接口 |
| 向量数据库 | Qdrant | 独立服务部署，支持向量检索、payload 过滤，具备生产化表达能力 |
| Embedding | OpenAI-compatible / BGE 可扩展 | 初期先接一种，代码层预留可插拔接口 |
| LLM | OpenAI-compatible API | 兼容 DeepSeek、智谱、通义、OpenAI 等模型服务 |
| 文档解析 | pypdf / python-docx / markdown | 按 MVP 阶段逐步支持常见文档格式 |
| 部署 | Docker Compose | 一键启动前端、后端、Qdrant 等服务 |

## 工程结构原则

- 后端不使用根目录单文件脚本，统一放在 `backend/app` 下。
- HTTP 接口按 `api/v1/endpoints` 拆分，为后续接口版本管理留空间。
- RAG 主链路放在 `services`，Qdrant、LLM、Embedding 等第三方调用放在 `integrations`。
- 请求和响应模型放在 `schemas`，避免路由层直接拼字典。
- 前端按 `features` 拆业务模块，知识库、文档、问答各自维护组件和状态。
- 基础设施配置放在 `infra`，本地脚本放在 `scripts`，项目文档放在 `docs`。

## MVP 0：最小可跑版

目标：用最短时间跑通 RAG 主流程，证明系统从上传文档到问答生成是闭环的。

### 功能范围

- FastAPI 后端服务。
- Qdrant 向量数据库接入。
- 支持上传 `txt` 或 `md` 文档。
- 文档切分、Embedding 生成、写入 Qdrant。
- 用户提问后执行 Top-K 向量检索。
- 调用 LLM 生成回答。
- 返回回答内容和引用片段。
- 提供 `.env.example`，配置模型地址、API Key、Qdrant 地址。

### 交付标准

- 后端可以本地启动。
- Qdrant 可以通过 Docker Compose 启动。
- 能通过 Swagger 或 curl 完成一次完整问答。
- README 中有最小启动步骤。

### 不做内容

- 不做复杂前端。
- 不做登录、权限、多租户。
- 不做异步队列。
- 不做混合检索。

## MVP 1：GitHub 简历展示版

目标：做成一个别人能看懂、能启动、能体验的完整项目。

### 功能范围

- React + Vite + TypeScript + Ant Design 前端。
- 知识库列表页面。
- 文档上传页面。
- 问答页面。
- 回答展示引用来源，包括文件名、片段内容、相似度分数。
- 支持 PDF、Markdown、TXT 至少 2-3 种格式。
- 支持删除文档或重建知识库索引。
- 后端提供 `/health` 健康检查接口。
- 后端输出基础结构化日志。
- Docker Compose 启动前端、后端、Qdrant。
- README 补充项目介绍、架构说明、启动方式、接口说明、演示截图位置。

### 推荐页面

1. 知识库管理页：展示知识库名称、文档数量、创建时间。
2. 文档上传页：选择知识库并上传文件，展示上传结果。
3. 智能问答页：输入问题，展示答案和引用来源。

### 交付标准

- 新用户按照 README 可以在本地启动项目。
- 可以通过 Web 页面完成上传和问答。
- GitHub 首页能清楚说明项目价值、技术栈和架构。
- 简历可以写成一个完整的前后端 RAG 项目。

## MVP 1.5：中小公司可用增强版

目标：从演示项目升级为轻量可落地项目，突出工程能力和业务适配能力。

### 功能范围

- 多知识库管理，例如企业制度库、产品手册库、客服 FAQ 库。
- 文档元数据管理：`kb_id`、`doc_id`、`filename`、`chunk_index`、`created_at`。
- API Key 简单鉴权。
- 对话历史记录。
- 上传状态管理：待解析、索引中、完成、失败。
- 文档列表、删除、重新索引接口。
- 混合检索：BM25 + Qdrant 向量检索。
- RRF 融合排序。
- 可选 rerank 能力。
- 基础错误处理和统一响应格式。

### 推荐亮点

- 用 Qdrant payload 实现知识库隔离和元数据过滤。
- 用混合检索提升错误码、单据号、专有名词等精确查询效果。
- 用引用来源降低大模型幻觉风险。
- 用 API Key 鉴权体现基础安全意识。

### 交付标准

- 可以面向中小企业内部知识库做轻量部署演示。
- README 中补充 Roadmap、常见问题和适用场景。
- 项目可以支撑简历中的“工程化 RAG 系统”描述。

## MVP 2：生产增强版 Roadmap

目标：作为后续路线图，不作为当前简历项目的强制交付。

### 可选能力

- 多租户隔离：`tenant_id` + `kb_id`。
- JWT 对接业务系统。
- Redis 管理会话历史。
- 异步任务队列处理大文档索引。
- 模型路由：按租户、场景或成本选择模型。
- 权限过滤：按角色、部门、文档级权限控制检索范围。
- Prometheus 指标和监控面板。
- Nginx 网关。
- Spring Boot 业务系统集成。
- 管理后台审计日志。

### 暂缓原因

这些能力更接近生产级企业项目，会显著增加开发周期。当前目标是快速完成一个能展示 RAG、前后端、向量检索和 Docker 部署能力的通用项目，因此 MVP 2 先作为后续规划即可。

## 推荐开发顺序

1. 搭建后端项目结构和配置管理。
2. 接入 Qdrant，完成 collection 初始化。
3. 实现文档上传、解析、切分。
4. 实现 Embedding 生成和向量入库。
5. 实现检索和 LLM 问答。
6. 增加引用来源返回。
7. 搭建 React + Ant Design 前端。
8. 完成上传、知识库、问答页面。
9. 补 Docker Compose 和 `.env.example`。
10. 完善 README、截图和演示说明。

## 简历表达建议

可以写为：

> 基于 FastAPI、Qdrant、React 和大模型 API 设计并实现面向中小企业的智能知识库问答系统，支持文档上传、向量化索引、语义检索、RAG 问答生成与答案来源引用，并通过 Docker Compose 提供一键部署能力。

增强版可以补充：

> 引入多知识库管理、文档元数据过滤、API Key 鉴权和混合检索策略，提升企业内部制度、产品文档、FAQ、错误码等知识检索场景下的可用性和准确率。
