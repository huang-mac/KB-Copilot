# MVP3 spec.md

> 本文件回答"要做什么"。只描述需求、用户故事和验收标准，不讨论具体技术方案。
>
> MVP3 的目标是把 KB Copilot 从"可用的 RAG 应用"升级为"可编排、可扩展、可观测、体验优秀的智能知识库问答底座"。
>
> 已有子 spec 直接引用，不在此重复：
> - SSE 流式输出：见 `.claude/specs/sse-streaming/spec.md`
> - 意图识别与工具调用：见 `.claude/specs/intent-routing/spec.md`

## 目标

MVP3 从两条线同时推进：

**基础设施线** — 补齐检索、存储、索引和可观测性底座，使系统具备混合检索、异步索引、MySQL 兼容和基础监控能力。

**体验线** — 从"一问一答"升级为"引导-理解-行动"的智能对话流，覆盖自动滚底、引用可视化、优雅降级、追问建议、响应式布局和暗色模式。

## 用户故事

### 检索增强

- 作为知识库使用者，我希望输入精确关键词（型号、编号、错误码）时能搜到准确结果，而不是仅靠语义近似匹配。
- 作为开发者，我希望可以按需开关 rerank，在精度和延迟之间做取舍。

### 异步索引

- 作为知识库使用者，我上传文档后不需要卡在页面上等待索引完成，可以继续做其他操作。
- 作为知识库使用者，我希望看到索引任务的实时状态（排队中 / 索引中 / 已完成 / 失败），失败时能看到原因。

### 体验基础

- 作为知识库使用者，我希望新消息出现时对话自动滚到底部，不需要手动拖动滚动条。
- 作为知识库使用者，我希望引用来源可以点击展开查看原文片段，证据用浅色背景高亮，增强可信度。
- 作为知识库使用者，当知识库搜不到相关内容时，我希望系统明确告知我"暂无相关信息"，并给出建议，而不是沉默或编造。
- 作为知识库使用者，我希望一键复制回答内容，方便粘贴到别处。
- 作为知识库使用者，我希望对回答给予"有用/无用"反馈，帮助改进服务质量。
- 作为知识库使用者，我希望删除不需要的历史会话，保持会话列表整洁。

### 体验升级

- 作为知识库使用者，我希望回答下方自动出现 2-3 个追问建议按钮，降低我继续提问的输入成本。
- 作为知识库使用者，对于常见问题（如价格、步骤），我希望以卡片、表格或步骤指示器展示，而不是大段纯文本。
- 作为知识库使用者，我希望拖拽 PDF/Word 文件就能上传，不需要限定 TXT/Markdown。
- 作为移动端使用者，我希望在手机上底部输入框固定，键盘弹起时对话自动滚到最新消息。
- 作为使用者，我希望支持暗色模式，在低光环境下使用不刺眼。

### 存储与监控

- 作为运维者，我希望可以选择 MySQL 作为元数据存储，以便应对后续的数据增长和多人访问。
- 作为运维者，我希望系统暴露基础监控指标（请求耗时、检索耗时、token 用量、错误数），以便排查问题和掌握运行状态。

## EARS 需求

### 混合检索

- 当用户提交问题时，系统应同时执行向量检索和关键词检索，并融合两路召回结果。
- 融合算法应使用 RRF（Reciprocal Rank Fusion），融合后保留各片段的召回来源标识（vector / keyword）。
- 当 `RERANK_ENABLED=true` 时，系统应对融合后的候选片段执行 rerank，并按新分数排序。
- 当 `RERANK_ENABLED=false` 时，系统应跳过 rerank，直接使用 RRF 融合结果。

### 异步索引

- 当用户上传文档时，系统应在 200ms 内返回索引任务信息（task_id + status=queued），不阻塞 HTTP 请求。
- 后台 worker 应异步执行文档解析 → 切分 → Embedding → Qdrant 入库流程。
- 用户可通过 `GET /api/v1/kbs/{kb_id}/index-jobs/{job_id}` 查询索引任务进度和结果。
- 当索引任务失败时，系统应记录失败原因（如"PDF 解析失败：文件损坏"）并可通过 API 查询。

### 对话体验

- 当新消息出现（用户发送或流式 token 更新）时，前端应自动将对话列表滚动到底部。
- 当用户主动向上滚动查看历史时，自动滚底行为应暂停；用户重新滚到底部后恢复。
- 当流式输出结束后，对话应停留在最底部，确保用户看到完整回答。
- 当用户删除会话时，系统应删除该会话及其所有关联消息，并从会话列表中移除。
- 当前正在查看的会话被删除后，系统应自动切换到下一个可用会话（如果存在）。
- 当用户点击引用标签时，系统应展开该引用的原文片段，并用浅色背景或左边框高亮。
- 当检索召回分数低于阈值或结果为空时，系统应返回"知识库中暂无相关信息，建议尝试换个问法或联系人工客服。"，不编造答案。
- 每条助手消息旁应提供"复制"按钮，点击后将完整 Markdown 原文复制到剪贴板。
- 每条助手消息旁应提供"有用/无用"反馈按钮，点击后提交反馈并确认。

### 追问建议

- 当助手回答完成后，系统应基于当前回答内容生成 2-3 个可能的追问建议。
- 追问建议以按钮形式展示在回答下方，用户点击后自动填入问题并发送。

### 结构化答案

- 对于包含步骤、价格、配置等结构化信息的回答，前端应以卡片/表格/步骤指示器等组件渲染（通过 Markdown 扩展或后端 type 标识）。

### 文件上传

- 上传组件应支持拖拽 PDF（.pdf）和 Word（.docx）文件。
- 后端应能解析 PDF 和 DOCX 文件内容，提取纯文本用于索引。

### 响应式布局

- 移动端（宽度 < 768px）输入框应固定在底部，历史会话用抽屉或折叠方式展示。
- 移动端键盘弹起时，对话区域应自动上推，最新消息保持在可见范围。

### 暗色模式

- 系统应支持亮色 / 暗色主题切换，默认为亮色。
- 暗色模式下所有组件（卡片、输入框、代码块、表格）应有对应的深色样式。

### MySQL 兼容

- 当 `METADATA_DB_PROVIDER=mysql` 时，文档、会话、消息、索引任务的存储应使用 MySQL。
- 默认 `METADATA_DB_PROVIDER=sqlite` 保持本地零依赖体验不变。
- 同一接口在两种存储下的行为应一致。

### 监控指标

- 系统应记录请求数、请求耗时、检索耗时、LLM 耗时、token 用量和错误数。
- 指标应通过 `GET /api/v1/metrics` 暴露（JSON 格式即可，MVP3 不做 Prometheus）。

## API 需求

```text
# 已有（行为可能调整）
POST   /api/v1/kbs/{kb_id}/chat          # 升级为意图驱动
POST   /api/v1/kbs/{kb_id}/chat/stream   # SSE 流式（已有 spec）
POST   /api/v1/kbs/{kb_id}/chat/{conv_id}/regenerate  # 已有 spec
POST   /api/v1/tools/order-query         # 已有 spec

# 新增
POST   /api/v1/kbs/{kb_id}/search        # 混合检索（独立可测）
GET    /api/v1/kbs/{kb_id}/index-jobs/{job_id}  # 查询索引任务状态
DELETE /api/v1/kbs/{kb_id}/conversations/{conv_id}  # 删除会话及其消息
POST   /api/v1/kbs/{kb_id}/feedback      # 提交回答反馈
POST   /api/v1/kbs/{kb_id}/suggestions   # 基于上下文生成追问建议
GET    /api/v1/metrics                   # 监控指标
```

## 数据需求

- `index_job`：`job_id, kb_id, doc_id, status (queued|processing|completed|failed), error_message, chunk_count, created_at, updated_at`。
- `feedback`：`message_id, rating (helpful|not_helpful), created_at`。
- `retrieval_source`：`source_type (vector|keyword), doc_id, filename, chunk_index, score, content`，用于标识召回来源。
- `search_result`：在 `Source` 基础上增加 `source_type` 和 `rerank_score`（如果启用 rerank）。
- `suggestions`：`["追问1", "追问2", "追问3"]` — 由 LLM 基于当前回答生成。

## 验收标准

### 检索增强
- 精确关键词（型号、错误码）的搜索结果比纯向量检索更准确（人工抽查 5 个 case）。
- RRF 融合结果中每个片段可追溯到来源（vector / keyword）。
- rerank 可通过配置开关启用/关闭，行为符合预期。

### 异步索引
- 上传文档后接口 200ms 内返回，不阻塞。
- `GET /api/v1/kbs/{kb_id}/index-jobs/{job_id}` 可查到任务状态变化。
- 索引失败时 `index_job.error_message` 包含失败原因。

### 对话体验
- 新消息出现时对话自动滚到底部。
- 用户向上滚动时暂停自动滚底，滚回底部后恢复。
- 无检索结果时显示明确提示，不编造。
- 引用标签可点击展开原文片段，证据区域有视觉区分。
- 复制按钮可将回答原文复制到剪贴板。
- 有用/无用反馈按钮可点击并提交。
- 删除会话后该会话从列表中消失，关联消息一并清除。
- 当前会话被删除后自动切换到下一个会话（如果存在），不存在则显示空状态。

### 追问建议
- 回答完成后下方出现 2-3 个追问建议按钮。
- 点击追问建议自动填入并发送。

### 文件上传
- 支持拖拽上传 PDF 和 DOCX。
- 上传的 PDF/DOCX 可被解析、索引，并在问答中被检索到。

### 响应式与暗色模式
- 移动端布局可用，输入框固定底部。
- 暗色模式切换正常，所有组件有对应的深色样式。

### MySQL 与监控
- `METADATA_DB_PROVIDER=mysql` 时 CRUD 行为与 SQLite 一致。
- `GET /api/v1/metrics` 返回有效指标数据。

## 不包含

- WebSocket 双向通信。
- 多工具并行调用或工具链。
- 实时订单系统对接（仅 mock）。
- 语言级代码高亮（highlight.js / Prism）。
- 断线重连自动续传。
- 对话导出 Markdown/PDF。
- 对话分享永久链接。
- iframe / Web Widget 嵌入。
- 用户自定义提示词或预设角色。
- 个性化记忆（用户偏好知识领域）。
- 多模态图片生成或展示（如回答中的图片内嵌）。
- API Key / JWT 鉴权。
- 多租户隔离。
- 文档级权限过滤。
- 管理员配置页。
- 真实 MCP 远程工具接入。
- 审计日志。
