# MVP1 spec.md：RAG 闭环演示版

> 本文件回答“要做什么”。只描述需求、用户故事和验收标准，不讨论具体技术方案。

## 目标

MVP1 应提供一个可本地启动、可通过 Web 页面演示的知识库 RAG 闭环。用户上传 Markdown 或 TXT 文档后，系统应能构建向量索引，并基于知识库内容回答问题和展示引用来源。

## 用户故事

- 作为演示用户，我希望上传 Markdown 或 TXT 文档，以便系统构建知识库索引。
- 作为演示用户，我希望输入知识库 ID 和问题，以便获取基于该知识库内容的回答。
- 作为演示用户，我希望看到回答的引用来源，以便追溯答案依据。
- 作为开发者，我希望在没有真实模型 API Key 时使用 mock 模式，以便完成本地冒烟测试。

## EARS 需求

- 当用户上传 `.md`、`.markdown` 或 `.txt` 文件时，系统应解析文档文本并构建可检索索引。
- 当用户上传不支持的文档格式时，系统应返回明确错误。
- 当用户上传空文档时，系统应返回明确错误。
- 当文档索引成功后，系统应允许用户基于同一 `kb_id` 发起问答。
- 当用户发起问答时，系统应按 `kb_id` 检索相关文档片段。
- 当检索到相关文档片段时，系统应基于片段内容生成回答。
- 当系统返回回答时，系统应同时返回引用来源。
- 如果配置为 mock 模式，系统应在不调用外部模型 API 的情况下完成上传和问答冒烟流程。

## API 需求

```text
GET    /api/v1/health
POST   /api/v1/kbs/{kb_id}/documents
POST   /api/v1/kbs/{kb_id}/chat
```

## 验收标准

- 用户可以通过 Web 页面上传 Markdown/TXT 文档。
- 上传成功后，文档会被解析、切分、向量化并写入向量库。
- 用户可以输入问题并获得基于文档内容的回答。
- 回答中包含引用来源。
- mock 模式下不依赖真实模型 API Key 也能完成冒烟测试。
- Docker Compose 能启动前端、后端和 Qdrant。

## 不包含

- 文档列表、删除、重新索引。
- 会话历史和多轮追问。
- 混合检索、RRF、rerank。
- 异步索引。
- 鉴权、多租户、权限过滤。
