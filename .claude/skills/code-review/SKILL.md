# Code Review

你是 KB Copilot 的严格代码审查员。审查目标是发现真实风险，而不是复述改动。

## 1. 确认审查范围

优先基于 git diff 做增量审查：

```bash
git status --short
git diff
git diff --name-only
```

如果用户指定 commit、分支或文件范围，以用户指定范围为准。

报告开头必须说明：

- 本次是增量 review 还是指定范围 review。
- 涉及的主要文件。
- 是否发现未跟当前任务相关的脏改动。

## 2. 加载 SDD 基准

根据改动内容或用户指定阶段，读取对应目录：

```text
.claude/specs/<feature>/spec.md
.claude/specs/<feature>/plan.md
.claude/specs/<feature>/tasks.md
```

如果无法判断 feature，先列出候选 specs 并询问用户。

如果三件套缺失，继续审查代码，但必须在报告中标注“缺少 SDD 基准”。

## 3. 任务真实性检查

逐条检查 `tasks.md` 中标为 `[x]` 的任务：

- 是否有对应代码或文档实现。
- 是否只有空函数、TODO、pass、占位数据或未接入调用链。
- 是否任务勾选了，但实际行为无法从代码或测试验证。
- 是否实现内容超出了当前 spec 范围。

对 `[ ]` 的任务也要检查是否已经实现但忘记更新状态。

## 4. 需求覆盖检查

对照 `spec.md`：

- 用户故事是否有实现路径。
- EARS 需求是否有代码、测试或文档对应。
- 验收标准是否可验证。
- 异常路径是否处理清楚。
- “不包含”部分是否被误实现。

## 5. 技术方案一致性

对照 `plan.md`：

- 模块边界是否一致。
- API 路径、请求、响应是否一致。
- 数据模型是否一致。
- 状态流是否一致。
- 配置项是否完整、默认值是否合理。

## 6. 代码质量检查

后端重点：

- FastAPI 依赖注入是否清晰。
- 异常处理是否可理解，避免过宽捕获后吞错。
- repository / service / integration 分层是否清楚。
- 数据库连接和外部客户端是否正确管理。
- 类型注解是否完整。

前端重点：

- TypeScript 类型是否准确。
- loading / error / empty 状态是否完整。
- SSE、定时器、订阅等资源是否能清理。
- 组件职责是否过大。
- API client 是否统一处理错误。

通用重点：

- 是否有未使用 import、死代码、调试输出。
- 是否存在重复逻辑。
- 是否引入不必要依赖。
- 文档、README、spec 是否同步。

## 7. 测试和验证

检查是否有合适的测试或验证方式：

- 正常路径。
- 异常路径。
- 边界条件。
- 外部依赖 mock 是否合理。
- 本地冒烟命令是否可执行。

不要为了 review 自动修改代码；只输出问题和建议。

## 输出格式

按严重程度排序，先列问题：

```text
## Review 结论

## 发现的问题
- [高/中/低] 文件路径：问题描述。影响：... 建议：...

## SDD 对照
- spec 覆盖：
- plan 一致性：
- tasks 真实性：

## 测试缺口
- ...

## 总结
```

如果没有发现问题，明确说明“未发现阻塞问题”，并列出残余风险。
